Enterprise-wide risk portfolio × Misiura (2015) × COSO IC/ERM × Safety-II × Resilience Engineering × DMDU

Архитектура operating model

Первый полный верхнеуровневый baseline-набор для авиакомпании. Не «абсолютная истина для всех», а минимальная operating model для старта. Совместим с QMS, SMS, compliance и ERM. Принцип: enterprise-wide portfolio view (Misiura, 2015), а не разрозненные silos. Дизайн учитывает требования COSO IC/ERM, RAG-потенциалы Hollnagel (Safety-II), адаптивные траектории DMDU и принцип антихрупкости.

Теоретический каркас

Kaplan & Mikes (2012, 2014)
Трёхклассовая типология: Preventable (eliminate), Strategy-execution (manage cost-effectively), External (reduce impact). Portfolio view.
Misiura (2015)
ERM framework: 5 компонентов (governance, process, portfolio, architecture, internal environment). Three lines of defence. Interdependencies. Context-sensitive design.
COSO IC (2013) / ERM (2017)
Principle 13: relevant quality information. Principle 16: design vs operating effectiveness. Entity-level portfolio view. Risk appetite → risk tolerances.
Safety-II / RE (Hollnagel)
4 RAG-потенциала: Respond, Monitor, Learn, Anticipate. Positive observations. Leading indicators. «Things that go right» как объект изучения.
DMDU / Adaptive pathways
Signposts = заранее определённые триггеры смены стратегии. Pre-committed responses. Optionality вместо предсказания.
Антихрупкость (Taleb)
Система усиливается от стрессоров: быстрая обратная связь, обучение, избыточность. Signposts = опционы. Event_Pressure = чувствительность к повторяемости.
Принцип иерархии рисков: верхний риск (Level 1) ≠ один уровень. Каждый декомпозируется на подриски (Level 2). Пример: S2 «Срыв программы ТО» ← P1, P3, P7, S8, E2. Верхний риск — управленческая проблема, подриски — причинная структура.

Формулы скоринга

1. Inherent risk score

Inherent_Score = Impact × Likelihood
// Impact = 1..5, Likelihood = 1..5, max = 25
// Speed НЕ входит в произведение — используется отдельно для Escalation_Priority
// Обоснование: COSO/ISO 31000 velocity of onset — квалификатор приоритизации, не magnitude

2. Control factor

Control_Strength = min(Design_Eff, Operating_Eff) // по ВСЕМ контролям риска из Table C

Strength | Factor
1 (отсутствует/не работает) → 1.00
2 (слабый) → 0.80
3 (средний) → 0.60
4 (сильный) → 0.40
5 (очень сильный) → 0.20

3. Residual score

Residual_Score = Inherent_Score × Control_Factor

4. Event pressure modifier

Event_Pressure = 1 + min(Repeat_Events / Threshold_Events × 0.25, 0.50)
// Ограничение сверху: max 1.50
// Если событий нет → Event_Pressure = 1.0 (нейтрально)
// Если события частые → до 1.5× от Residual

5. Dynamic residual score

Dynamic_Residual = Residual_Score × Event_Pressure
// Риск чувствителен к повторяемости событий — антихрупкость наоборот:
// система «слышит» нарастающее давление через event stream

6. Escalation priority

Escalation_Priority = Speed × Dynamic_Residual
// Speed = 1..5 (velocity of onset)
// Два риска с одинаковым Dynamic_Residual, но разным Speed →
// быстрый риск обрабатывается первым
// Используется для СОРТИРОВКИ, не для оценки magnitude

7. Appetite status

// Стартовые пороги (калибруются после 3 мес.):
Dynamic_Residual 0–5 → Within appetite
Dynamic_Residual 6–12 → Near appetite
Dynamic_Residual >12 → Exceeded appetite

// После 3 месяцев с реальными данными:
// P75 от распределения Dynamic_Residual = граница yellow
// P95 = граница red
// Это устраняет произвольность стартовых порогов

Полный пример расчёта: S2 — Срыв программы ТО

Impact = 4, Likelihood = 4
Inherent = 4 × 4 = 16

Controls: CTRL-21 (weekly review): Design=4, Operating=2 → min=2
Control_Strength = 2 → Factor = 0.80
Residual = 16 × 0.80 = 12.8

Repeat_Events=4, Threshold=8
Event_Pressure = 1 + min(4/8 × 0.25, 0.50) = 1 + 0.125 = 1.125
Dynamic_Residual = 12.8 × 1.125 = 14.4 → Exceeded

Speed = 3
Escalation_Priority = 3 × 14.4 = 43.2

Потоки данных между таблицами

Пять связанных потоков. Таблицы не автономны — они образуют замкнутую систему обратной связи.
Поток 1: Events → Risks (снизу вверх)
Table B → Table A. Каждое событие привязывается к ≥1 риску. Events > threshold → risk review. Repeat/systemic → пересмотр score. Severity ≥4 → немедленная эскалация. Новый тип → возможен Level 2 sub-risk.
Поток 2: Risks → Actions (сверху вниз)
Table A → Table D. Exceeded и Near appetite → обязательная mitigation/CAPA. Каждый top risk: owner + indicators + controls + review cycle. Risk без actions = governance weakness.
Поток 3: Controls → Residual score
Table C → Table A. Control_Strength = min(D,O). Weakness_Flag → action в Table D. Control failed → немедленный пересмотр risk score.
Поток 4: Metrics → Escalation
Table F → Table E. Green → 1-я линия. Yellow → 2-я линия. Red → executive + review forum. Speed определяет приоритет внутри каждого уровня.
Поток 5: Interdependencies → Cascade assessment
Table A (Linked_Risks). При пересмотре любого риска → проверить ВСЕ linked risks. Если S2 ухудшился → проверить S3, S1, E8, P7, P1. Реализация «combined effects of multiple scenarios» (Misiura, Alpha FRM framework). Каскадный пересмотр — не опция, а обязательный шаг.

Карта субъективности в скоринге

Каждая точка принятия решений — либо data-driven, либо expert judgment. Цель: не устранить экспертизу, а сузить коридор произвольности через данные. Стартовая система неизбежно субъективна. За 6–12 месяцев — перевод ключевых компонентов из «субъективных» в «полу-субъективные».

Правило: данные не заменяют экспертизу — они ограничивают пространство для произвольных оценок. Чем больше data-driven компонентов, тем у́же коридор субъективности. Event_Pressure — единственный полностью data-driven компонент на старте. Цель за 6–12 месяцев: перевести Impact, Likelihood и Control effectiveness из «субъективных» в «полу-субъективные».

Signposts для adaptive pathways (DMDU)

Signpost = заранее определённый триггер, при котором меняется стратегия управления. Не «реагируем когда случилось», а «если X достигнет Y — переключаемся на план B». Каждый signpost = опцион по Taleb: стоимость мониторинга триггера минимальна, выигрыш при срабатывании — быстрое переключение вместо паники.

Рекомендация по старту

Не строить огромную систему сразу. Минимальный стартовый комплект:
• 12–15 top risks (самые критичные из 26)
• 8 ключевых метрик (M-01..M-08)
• 1 единый event log (все типы событий в одной таблице)
• 1 таблица actions/CAPA
• 1 таблица controls с Design_Eff + Operating_Eff
• 3 signposts (SP-1, SP-2, SP-5)
• Ежемесячный review

Этого достаточно, чтобы система начала жить. Системные метрики (M-10, M-11) добавляются после первого квартала. Расширение — после 2–3 циклов с реальными данными и калибровки thresholds.
